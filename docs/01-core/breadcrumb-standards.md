## 2. WAI-ARIA Authoring Practices Guide (APG)

### ✅ Pattern Compliance

Following [W3C ARIA Breadcrumb Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/breadcrumb/):

| ARIA Requirement | Implementation | Notes |
|------------------|----------------|-------|
| **Navigation Landmark** | `<nav>` element | Identifies breadcrumb as navigation region |
| **Landmark Label** | `aria-label="Breadcrumb"` | Descriptive label for assistive technologies |
| **Current Page Indicator** | `aria-current="page"` on last link | Indicates current location (on `<a>`, not `<span>`) |
| **Semantic List Structure** | `<ol>` ordered list | Conveys hierarchical relationship |
| **Visual Separator Handling** | CSS `::after` pseudo-element | Prevents screen reader announcement |

### Critical W3C Compliance Fix

**Per W3C spec**, the current page should be marked with `aria-current="page"` on an **anchor element**, not a span:

```tsx
// ✅ CORRECT (W3C Standard)
<Link href={crumb.href} aria-current="page" onClick={(e) => e.preventDefault()}>
  <span itemProp="name">{crumb.label}</span>
</Link>

// ❌ INCORRECT (Previous implementation)
<span aria-current="page">{crumb.label}</span>
```

**Why this matters:**
- Screen readers expect `aria-current` on links
- Maintains consistent navigation semantics
- Better cross-browser compatibility
- Follows established ARIA patterns

## 3. Schema.org Structured Data

### ✅ Dual Implementation

The breadcrumb component provides structured data in **two formats**:

#### A. Microdata (Embedded in HTML)

```tsx
<nav itemScope itemType="https://schema.org/BreadcrumbList">
  <ol>
    <li itemProp="itemListElement" itemScope itemType="https://schema.org/ListItem">
      <Link href="/" itemProp="item">
        <span itemProp="name">Home</span>
      </Link>
      <meta itemProp="position" content="1" />
    </li>
  </ol>
</nav>
```

**Benefits:**
- Visible to search engines in HTML
- No JavaScript required
- Works with older crawlers
- Validates with Google Rich Results Test

#### B. JSON-LD (in `<head>`)

Generated by `SchemaFactory.ts`:

```json
{
  "@type": "BreadcrumbList",
  "@id": "https://z-beam.com/materials/metal/aluminum#breadcrumb",
  "itemListElement": [
    {
      "@type": "ListItem",
      "position": 1,
      "name": "Home",
      "item": "https://z-beam.com/"
    },
    {
      "@type": "ListItem",
      "position": 2,
      "name": "Metal",
      "item": "https://z-beam.com/materials/metal"
    },
    {
      "@type": "ListItem",
      "position": 3,
      "name": "Aluminum",
      "item": "https://z-beam.com/materials/metal/aluminum"
    }
  ]
}
```

**Benefits:**
- Preferred by Google
- Easy to validate
- Centralized schema management
- Consistent with other structured data

### Why Both Formats?

| Format | Use Case |
|--------|----------|
| **Microdata** | Legacy crawlers, older browsers, direct HTML parsing |
| **JSON-LD** | Modern search engines (Google, Bing), rich results |

This **belt-and-suspenders approach** ensures maximum SEO coverage.

## 4. Semantic HTML5

### ✅ Proper Semantic Structure

```html
<nav>          <!-- Identifies navigation region -->
  <ol>         <!-- Ordered list (hierarchy matters) -->
    <li>       <!-- List items for each breadcrumb -->
      <a>      <!-- Links to pages (including current) -->
        <span> <!-- Text content (for microdata) -->
```

**Why `<ol>` instead of `<ul>`?**
- Breadcrumbs represent an **ordered hierarchy**
- Position matters (Home → Category → Item)
- Semantic correctness for assistive technologies

**Why `<a>` for current page instead of `<span>`?**
- W3C ARIA specification requirement
- Consistent navigation semantics
- Better screen reader compatibility
- `onClick={(e) => e.preventDefault()}` prevents navigation

## 5. Visual Separator Best Practice

### ✅ CSS Pseudo-elements (Not Inline SVG)

**Previous Implementation (Problematic):**
```tsx
{index > 0 && (
  <svg className="w-3 h-3 text-gray-400 mx-1" aria-hidden="true">
    {/* SVG path */}
  </svg>
)}
```

**Problems:**
- Screen readers may announce presence of graphics
- Extra DOM nodes
- Less performant
- Harder to customize

**Current Implementation (Optimized):**
```css
.breadcrumb-item:not(:last-child)::after {
  content: '/';
  margin: 0 0.5rem;
  color: rgb(156, 163, 175);
  display: inline-block;
}
```

**Benefits:**
- Not in accessibility tree (never announced)
- Pure CSS (no extra DOM nodes)
- Easy theming/customization
- W3C recommended approach

## 9. 3-Tier Priority System

### ✅ Intelligent Breadcrumb Generation

```typescript
Priority 1: Explicit frontmatter breadcrumb array
Priority 2: Auto-generate from category/subcategory/name
Priority 3: URL parsing fallback
```

**Implementation:**
```typescript
export function Breadcrumbs({ breadcrumbData }: BreadcrumbsProps = {}) {
  const pathname = usePathname();
  
  let allBreadcrumbs: BreadcrumbItem[];
  
  if (breadcrumbData && breadcrumbData.length > 0) {
    // Priority 1: Use frontmatter data
    allBreadcrumbs = breadcrumbData;
  } else {
    // Priority 2/3: Generate from URL
    allBreadcrumbs = generateBreadcrumbsFromUrl(pathname);
  }
  
  return (/* ... */);
}
```

**Centralized Generation** (`app/utils/breadcrumbs.ts`):
```typescript
export function generateBreadcrumbs(
  frontmatter: Partial<ArticleMetadata>,
  pathname: string | null
): BreadcrumbItem[]
```

This utility is used by:
1. `Breadcrumbs` component (visual UI)
2. `SchemaFactory` (JSON-LD structured data)
3. Future components (DRY principle)
